# 生产环境构建实践清单

这篇不是讲工具 API，而是讲线上项目最常用、最容易踩坑的构建原则。

## 1. 先明确目标，不要只盯着“包大小”

生产环境通常同时追求 4 件事：

- 首屏关键资源尽可能小
- 二次访问尽量命中缓存
- 发布过程不出现资源错配
- 线上报错能定位、能回滚

所以一个“好构建”不等于 `dist` 更小，而是：

- 首页 chunk 更合理
- 缓存更稳定
- 监控和排障链路完整
- 新版本与旧页面共存时不炸

## 2. 长期缓存是第一优先级

### 建议策略

- HTML：`no-cache` 或短缓存
- JS / CSS / 图片 / 字体：文件名带 hash，走长缓存
- 保留最近几版静态资源，不要发布后立刻清理旧文件
- 服务端通过 manifest 或 HTML 模板注入当前版本资源

### 为什么很多项目发布后会报 `ChunkLoadError`

典型原因：

1. 用户打开了旧 HTML。
2. 页面运行时要去加载旧 chunk。
3. 你在 CDN 或静态服务器上把旧 chunk 删掉了。
4. 浏览器请求到 404，页面崩溃。

所以发布不是“替换文件”这么简单，而是“让旧页面还能找到它依赖的旧资源”。

## 3. 代码分割要围绕用户路径，而不是围绕文件夹

### 推荐拆法

- 路由级拆包：每个页面按需加载
- 重依赖单独拆：图表、富文本、PDF、地图、代码编辑器
- 框架层拆包：`react` / `vue` 及其核心生态
- 高频更新与低频更新依赖分开：避免一个业务改动带着全部 vendor 失效

### 常见反模式

- 所有三方库都塞进一个 `vendor.js`
- 每个组件都强行动态导入，造成几十个小 chunk
- 只看总包体积，不看首屏 waterfall

判断标准不是“chunk 越多越高级”，而是：

- 首屏少拉了多少不必要资源
- 重依赖有没有被延迟到真正使用时
- 关键路径上的请求数量是否失控

## 4. Tree-shaking 不是自动生效的

它一般依赖这些前提：

- 依赖包提供 ESM 版本
- 没有把模块整体 `import * as xxx`
- `package.json` 里的 `sideEffects` 配置正确
- 构建工具知道哪些导出未被使用

常见失效场景：

- 引入的是 CommonJS 包
- 组件库入口文件带了全量副作用
- 工具函数写法导致静态分析失败

所以优化时要先看“依赖本身能不能摇掉”，再看压缩器选项。

## 5. CSS 也是构建优化重点

### 生产环境常见策略

- 提取 CSS 为独立文件，便于缓存
- 页面级样式跟随路由拆分
- 大型设计系统控制全局样式膨胀
- 谨慎做 critical CSS，收益不稳定时不要过度复杂化

### 常见问题

- 一个全局 `index.scss` 越积越大，任何页面都被迫加载
- 样式拆分后顺序变化，导致覆盖关系异常
- 原子化 CSS / CSS Modules / scoped 样式混用后难以治理

## 6. Source Map 不只是调试体验问题

生产环境推荐思路：

- 生成 hidden source map
- 上传到错误监控平台
- 产物和 source map 绑定版本号
- 不直接对公网暴露完整源码映射

否则你会遇到两种极端：

- 完全没 map，线上报错只能看压缩后的乱码栈
- 直接公开 map，源码泄露成本过低

## 7. 环境变量和密钥不是一回事

前端环境变量的本质是“构建时注入”。

这意味着：

- `VITE_*`、`process.env.*` 最终都可能进到浏览器产物
- 只要用户能下载 JS，就可能看到这些值
- 真正的密钥应该只存在服务端

另外还要注意：

- `base` / `publicPath` 必须与部署目录一致
- `staging` 和 `prod` 不应共用同一套资源前缀
- 静态资源域名切换时要检查字体、图片、跨域头

## 8. 重依赖要分类型治理

### 可以考虑拆出来按需加载的

- `echarts`
- `monaco-editor`
- `pdfjs-dist`
- 地图 SDK
- 大型富文本编辑器

### 可以考虑外链的

- 极少变更、稳定、公共 CDN 命中率高的基础库

### 不要轻易外链的

- 版本经常变化的业务依赖
- 强耦合于构建产物的框架运行时代码
- 需要私有部署、内网环境、离线环境的资源

外链本质上是把“包体积”换成“运行时网络依赖”，不是无成本优化。

## 9. Worker 的处理方式要看体积和复用方式

### 内联 Worker

- 不额外发请求
- 适合很小的 worker
- 缺点是会把主包一起变大

### 独立 Worker 文件

- 首次使用时单独请求
- 更利于缓存和多页面复用
- 适合 PDF、音视频、复杂解析等重逻辑场景

生产里通常优先独立文件，而不是一律内联。

## 10. 构建性能优化要区分“开发慢”还是“CI 慢”

### 开发阶段常见抓手

- Vite 的依赖预构建
- Webpack 文件系统缓存
- Babel / TS 编译缓存
- 减少不必要 loader 链路

### CI 阶段常见抓手

- 缓存 `node_modules` 或包管理器 store
- 缓存构建缓存目录
- 分离类型检查、lint、单测、打包
- 对分析产物做增量治理，不要每次手工排查

## 11. 包体积分析要看 3 个维度

1. 首屏入口有多大。
2. 大依赖是否进了首屏。
3. 是否存在重复依赖或错误公共包。

一些常见问题：

- 同时打进了 `lodash` 和 `lodash-es`
- 同一 UI 库出现两个版本
- `moment` / 图表库被首页直接引用
- 业务代码改一行，vendor hash 整体变化

## 12. 发布与回滚建议

推荐做法：

- 资源文件名带 hash
- HTML 短缓存
- 保留最近几版静态资源
- source map 单独上传并和版本绑定
- 部署系统记录版本号、构建号、git commit

如果要灰度：

- 不同版本资源不要共用会互相覆盖的文件名
- 同一个 HTML 模板要只引用同版本资源
- 回滚时优先回滚 HTML 与资源映射，不只是回滚后端接口

## 13. 线上高频故障排查表

| 现象 | 常见原因 | 优先排查 |
| --- | --- | --- |
| `ChunkLoadError` | 旧页面请求旧 chunk，旧资源被删 | 是否保留历史资源、HTML 是否短缓存 |
| 页面白屏但本地正常 | `base/publicPath` 错误、静态资源跨域、CDN 路径问题 | 资源请求 URL、控制台报错、Nginx/CDN 配置 |
| 包很大但看不出原因 | 大依赖进入首屏、重复依赖、拆包失效 | bundle analyzer、依赖树 |
| Tree-shaking 没效果 | CJS、错误副作用声明 | 依赖格式、`sideEffects`、导入方式 |
| 样式异常 | CSS 顺序变化、代码分割后加载时机变化 | 提取顺序、异步页面样式 |
| 线上报错无法定位 | 没有 source map 或版本不匹配 | sourcemap 上传流程、release 标识 |

## 14. 一份实用检查清单

上线前至少确认：

- 首页和核心路由都做了按需加载
- 重依赖没有误进首屏
- 资源文件名带 hash
- HTML 不做长缓存
- 发布后保留旧静态资源
- source map 已上传且版本一致
- `base/publicPath` 与部署路径一致
- 产物分析报告已看过一遍
- 环境变量里没有真正的服务端密钥

如果这 9 条没有问题，构建链路通常已经超过“只会配脚手架默认配置”的水平。
